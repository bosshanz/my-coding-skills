# Progressive Admission Correction

## Problem

The first Loop Engineering v2 revision could be read as if every new requirement should enter `clarify`, and every sufficiently complex clarification should become a Loop.

That is not the intended operating model.

## Corrected Routing

```text
clear bounded task -> dev
material ambiguity -> targeted dev questions or explicit clarify
stronger independent judgment -> acceptance
persistent or repeated feedback problem -> governed Loop
```

## Key Boundaries

- `clarify` is not mandatory for every requirement.
- A clarification conversation does not automatically become a Discovery Loop.
- A Discovery Loop does not automatically require a workspace.
- Multi-file, multi-step, planned, tested, or important work is not sufficient Loop admission.
- High risk may justify stronger acceptance without requiring persistent Loop governance.
- Loop artifacts are created only when state, feedback, evaluation, or decisions must persist across independent iterations.

## Design Goal

Use the lowest process level that can safely solve the problem. Escalate only when evidence shows that the current level is insufficient.
