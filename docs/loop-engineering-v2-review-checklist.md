# Loop Engineering v2 Review Checklist

- [ ] The shortest reliable route was evaluated before proposing a Loop.
- [ ] Ordinary requirements and bugs remain in `dev` by default.
- [ ] Top-level `clarify` is used only for explicit interviews or material ambiguity.
- [ ] A clarification conversation does not automatically create a Loop.
- [ ] Multi-file, multi-step, planned, tested, or important work is not treated as sufficient Loop admission.
- [ ] The Loop reduces more complexity than its governance artifacts introduce.
- [ ] Discovery and Delivery are clearly separated where a governed Loop is actually needed.
- [ ] Every admitted Loop has red-capable evidence.
- [ ] Verifier owner, version, blind spots, and review cadence are explicit only where warranted.
- [ ] The Maker cannot silently redefine success criteria.
- [ ] Acceptance reports requested and achieved independence.
- [ ] Stronger acceptance can be used without forcing a persistent Loop.
- [ ] Human-only and irreversible decisions are visible.
- [ ] `CHECKPOINT.md` is used only when future runs need preserved rationale.
- [ ] Multi-Loop repositories can use stable named directories.
- [ ] `clarify`, `dev`, and `acceptance` share the same context model without becoming mandatory phases.
- [ ] New references are checked by `skills-doctor.sh`.
